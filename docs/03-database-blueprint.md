# Supabase Database Blueprint

## Status And Scope

This is the authoritative logical schema for Phase 1 design. It is not SQL and does not authorize migration creation. The existing `0001_core_learning_platform.sql` predates this blueprint and must be reconciled through an approved migration plan.

## Conventions

- PostgreSQL `public` holds application tables; Supabase owns `auth` and `storage` schemas.
- Primary keys are `uuid` generated by PostgreSQL. External/provider IDs are text and never become primary keys.
- Tenant-owned tables carry `organization_id uuid not null`; platform-global rows are explicitly identified.
- Mutable aggregates use `created_at timestamptz`, `updated_at timestamptz`, and where needed `archived_at timestamptz`. Append-only facts use `occurred_at`/`created_at` only.
- Human-readable slugs are unique within their owner/version scope, not globally unless explicitly public.
- Money uses integer minor units plus ISO currency. Durations use integer seconds. Scores use constrained numeric values.
- JSONB is reserved for versioned provider payloads, flexible content bodies and non-relational metadata; identities, ownership, status and authorization fields remain relational.
- Sensitive object bytes live in private Supabase Storage buckets. Database rows store bucket/object metadata and classification.
- Soft deletion is not a privacy erasure mechanism. Erasure/anonymization is executed by controlled privacy jobs and recorded as evidence.
- High-volume append-only tables are time partition candidates: audit events, analytics events, AI usage, notification deliveries, webhook deliveries and integrity events.

### Classification

| Code | Meaning                                                       |
| ---- | ------------------------------------------------------------- |
| P0   | Public/non-personal                                           |
| P1   | Internal operational data                                     |
| P2   | Personal data                                                 |
| P3   | Sensitive personal, assessment, employment or behavioral data |
| P4   | Secret/credential material; store only hashed or encrypted    |

### Retention Classes

| Code | Default rule; tenant/jurisdiction policy may shorten unless legally required     |
| ---- | -------------------------------------------------------------------------------- |
| R1   | Account/tenant lifetime plus 30 days, then erase or anonymize                    |
| R2   | Learning record lifetime plus 3 years                                            |
| R3   | Credential lifetime plus 7 years after expiry/revocation                         |
| R4   | Security/compliance evidence 7 years, append-only; legal hold overrides          |
| R5   | Transaction/tax records 8 years                                                  |
| R6   | Notification/integration operations 90 days; payloads 30 days                    |
| R7   | AI content 30 days by default; usage metadata 13 months; configurable by purpose |
| R8   | Analytics raw events 13 months; aggregates 3 years with privacy thresholds       |
| R9   | Until revoked/expired plus evidence under R4                                     |
| R10  | Configuration/version history for platform lifetime                              |

`CASCADE` below means lifecycle cascade within the same tenant aggregate. Compliance-critical facts instead use `RESTRICT` or actor nullification so history survives account deletion.

In the catalog, an unqualified conventional foreign key resolves as follows: `organization_id -> organizations.id`, `profile_id` and actor profile fields `-> profiles.id`, `storage_object_id -> storage_objects.id`, and every other `<entity>_id -> <entities>.id` unless an explicit target or self-reference is shown. Migration review must expand every shorthand into a named foreign-key constraint and index it where it is not the leading column of an existing index.

## Public Verification Design

Anonymous clients never select `certificates`. A rate-limited server route calls a narrowly scoped PostgreSQL function with the full high-entropy verification token. The function returns only holder display name, achievement, issuer, issue/expiry dates and current status from `certificate_public_views`. Tokens are not searchable, listable or logged in plaintext.

## Immutability And Audit Enforcement

- Revoke `UPDATE` and `DELETE` from application roles on audit, consent event, PII reveal event, certificate status event, payment event, assessment integrity event and application stage event tables.
- Inserts occur through security-definer functions that validate actor, tenant, purpose and transition, set server timestamps, and reject caller-supplied actor identity.
- Service-role possession is not a human-admin capability. Operational access is isolated, time-bound and monitored.
- Corrections append compensating events. Retention execution may partition-drop or cryptographically erase only under approved policy and legal-hold checks, while preserving a non-identifying execution manifest.
