# Master Enum Catalog

## Storage Policy

These are logical controlled domains. Use native PostgreSQL enums only for values that are extremely stable and migration-safe. Use checked text for bounded but evolving lifecycle values, lookup/registry tables for tenant-extensible roles/models/tasks, and external standards for language/country/timezone/currency. Unknown values fail validation; no free-form status strings enter production.

`Extensibility` records the only permitted expansion path. Removing or reinterpreting a value is breaking and requires data migration plus ADR.

## Enum Change Rules

- Additive checked-text values require application compatibility, state transitions, analytics mapping and event schema review.
- Native enum addition requires a dedicated migration and deployment-order review; removal/rename is prohibited without replacement/backfill.
- Registries use stable keys; labels/localizations are mutable metadata.
- Status history stores old values permanently even after UI deprecation.
- Provider/model/event values are never native enums because their lifecycle is operationally dynamic.
